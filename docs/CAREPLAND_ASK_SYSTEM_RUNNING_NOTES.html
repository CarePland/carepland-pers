How It Works Test And Validation Items Early Review Questions

Part 1: Natural-Language Explanation

Ask is CarePland's unified place for users to say what is on their mind without first deciding what kind of request they are making.

A user might ask a how-to question, report something confusing, describe a bug, suggest a feature, mention a workflow problem, or raise an account/access concern. The user-facing experience should not make them choose a category before typing. They should simply see Ask, write naturally, and feel that the system is listening respectfully.

The Ask workflow starts with conversation, not ticketing. The panel is meant to feel lightweight, warm, and safe: more open dialogue than support queue. The assistant may ask a brief clarifying question when that would materially improve the answer or the routing decision. If the conversation reaches a useful stopping point, the panel acknowledges the submission and closes the input area.

Behind the scenes, Ask is an intake system. It stores the conversation as a thread with messages. When the conversation is ready to route, CarePland creates an Ask submission snapshot containing the original wording, transcript, AI summary, router category, confidence, rationale, safety flags, and recommended next actions.

The router is intentionally modular. It should classify and route rather than try to be every assistant at once. As the system matures, separate interpreters can handle support answers, wishlist/feature interpretation, bug/friction interpretation, workflow feedback, account/access issues, and limited off-topic handling. Each path should have its own prompt/instructions and its own output shape so it can be improved independently.

The first downstream interpreter passes are wired for feature/workflow and bug/friction routes. The router still decides the broad path, then the relevant interpreter enriches the intake with cleaner structured recommended actions for Admin review. For feature/workflow items, that includes the suggested feature or workflow, pain point, desired outcome, affected app area, and urgency clues. For bug/friction items, that includes what the user tried, what they expected, what happened instead, reproducibility clues, affected area, and whether the issue may be usability confusion rather than a product defect.

The Clarifier is now a separate runtime module. When the router thinks clarification may help, The Clarifier decides whether one more question is actually useful. It can return a warm follow-up question, or it can stop the questioning and provide a short understanding summary for review. Clarification limits still set the ceiling, but the module helps avoid asking questions simply because the system can.

The off-topic handler is also a bounded runtime module. If something is harmlessly outside CarePland's purpose, it can close the loop with a short redirect. If an off-topic message carries possible medical, emergency, privacy, security, account-access, abuse, or data-loss risk, it should send the item to review instead of closing it casually.

Admin review happens in Support > Ask - Review. The review surface is decision-first. It shows a concise recommended action before raw data, with the longer transcript, router rationale, JSON recommendation details, safety flags, and manual review note available in collapsible sections. This keeps the page useful without forcing every review to feel like reading a case file.

Ask outputs are cross-referenceable. A single Ask conversation can lead to a support ticket, Product Management item, wishlist item, bug item, workflow item, or later related records. The intake item remains the source record, and downstream items should link back to it. This preserves the original user language and avoids losing context when the same conversation touches more than one area.

Support-question outcomes are now part of the review workflow. If the assistant answered well, Admin can accept the answer and close the Ask item. If human follow-up is needed, Admin can create a linked support ticket from the Ask item. If the recommendation was wrong, Admin can reject or override it. These decisions are not just workflow bookkeeping; they are model-quality feedback.

Recommendation decisions should become a learning loop. Over time, CarePland should be able to see where Admin agrees with the AI, where Admin disagrees, which categories are weak, which prompts need refinement, and when confidence thresholds are trustworthy enough to allow more automation. Rejected or overridden items should remain available for analysis rather than being erased.

Ask also logs estimated AI operation costs through a reusable cost-tracking pattern. This is admin-only operational information. It should help answer practical questions such as which AI workflows are expensive, how cost changes as usage grows, and whether future automation decisions are financially reasonable.

Ask Review includes a non-destructive module lab for prompt and routing evaluation. Admin can paste a batch of test questions, choose one Ask module, and review compact summaries, confidence, quality notes, and raw structured output. These lab runs do not create Ask submissions or downstream work items, but they do log estimated AI cost by module.

The assistant should not pretend to be human or deny that it is AI. At the same time, the user-facing product should not foreground AI unnecessarily. User-facing Ask module responses should also avoid first-person assistant phrasing such as I, me, my, we, we're, we've, and we'll whenever practical. The goal is to make Ask sound like a CarePland routing surface rather than a human agent. This global rubric is Admin-visible in AI Prompts as Ask user response rubric.

Part 2: Test And Validation Items

These items are meant to be revisited after real Ask data exists. They are not all launch blockers.

Basic User Experience

  • Confirm the Ask button appears app-wide for signed-in users where expected.
  • Confirm the panel opens and closes consistently, including outside-click close.
  • Confirm closing the panel resets the conversation state.
  • Confirm the guidance text comes from Dynamic Text.
  • Confirm the acknowledgement text comes from Dynamic Text.
  • Confirm the duplicate-submission acknowledgement comes from Dynamic Text.
  • Confirm the panel does not feel like a ticket form: no ticket number, no forced category, no overly robotic labels.
  • Confirm the input disappears after the conversation is concluded.
  • Confirm messages have comfortable spacing and readable contrast on mobile and desktop.

Conversation Behavior

  • Test a straightforward how-to question that can be answered immediately.
  • Test a question with a typo where a warm clarification is appropriate.
  • Test a vague question that needs one or two follow-up questions before routing.
  • Confirm The Clarifier asks one useful question when the router recommends clarification.
  • Confirm The Clarifier can stop questioning and summarize when another question would be low value.
  • Test a conversation that reaches the clarification limit and is sent for review with a brief summary.
  • Test whether the assistant asks useful questions rather than unnecessary ones.
  • Test whether the assistant stays in the background as AI unless asked directly.
  • Test direct questions such as Are you AI? and confirm the answer is transparent but not overexplained.
  • Test harmless off-topic input and confirm it receives a brief redirect.
  • Test risky off-topic input and confirm it is routed for review rather than closed.

Duplicate Handling

  • Submit the same message twice in one panel session.
  • Submit the same message again after closing and reopening the panel.
  • Submit the same message from another page.
  • Confirm duplicates are acknowledged warmly and do not create unnecessary duplicate review work.
  • Review whether the duplicate window is too strict or too loose after real usage.

Routing Quality

  • Collect examples of support_question, bug_report, feature_request, workflow_feedback, account_or_access_issue, unclear_or_needs_human_review, and off_topic.
  • Compare router category to Admin judgment.
  • For feature/workflow routes, check whether the interpreter preserves the user's idea without over-formalizing it.
  • For bug/friction routes, check whether the interpreter clearly separates tried, expected, and actual behavior.
  • Track confidence levels for correct and incorrect routes.
  • Look for categories that are frequently confused with each other.
  • Look for confidence scores that are too high on weak recommendations.
  • Review whether unclear_or_needs_human_review is used appropriately or becomes a dumping ground.

Ask Module Lab

  • Paste a batch of test questions and confirm the chosen module returns one compact result per line.
  • Confirm module-lab runs do not create Ask submissions, support tickets, bugs, workflow items, or wishlist items.
  • Compare the router output against expected categories before changing production prompts.
  • Use The Clarifier lab to test whether follow-up questions feel useful, brief, and conversational.
  • Use interpreter labs to check whether structured outputs preserve the user's original intent without over-formalizing it.
  • Confirm raw structured output is available for deeper inspection without dominating the first result view.
  • Confirm module-lab AI cost appears under module-specific operation keys in Admin cost tracking.

Admin Review Workflow

  • Confirm new Ask submissions appear in Support > Ask - Review.
  • Confirm Ask routing settings load and save from Support > Ask - Review.
  • Confirm default clarification turns cannot exceed the absolute maximum after saving.
  • Confirm active/closed/category filters behave as expected.
  • Confirm the recommended action is readable without expanding raw details.
  • Confirm each section expands independently.
  • Confirm Expand all and Collapse all work for the selected submission.
  • Confirm long transcripts remain readable and do not dominate the first view.
  • Confirm review notes and routing-state changes save correctly.

Decision Actions

  • Accept an answered support question and confirm the Ask item closes.
  • Create a support ticket from an Ask item and confirm the ticket appears in Support > Tickets.
  • Confirm the created ticket links back to the Ask submission.
  • Confirm the Ask submission links to the created support ticket.
  • Confirm related support tickets and Product Management items appear on the Ask review item.
  • Confirm related-item buttons move Admin to the expected downstream review area.
  • Confirm support tickets created from Ask show a link back to the source Ask review item.
  • Confirm Product Management items created from Ask show a link back to the source Ask review item.
  • Create Product Management items from bug, wishlist, and workflow recommendations.
  • Reject a recommendation and confirm it appears in recommendation-decision stats.
  • Mark an override and confirm it appears separately from accepted/rejected decisions.
  • Confirm accepted decisions cannot accidentally create duplicate downstream records from the same recommendation.

Model-Quality Feedback

  • Review agreement rate after the first meaningful batch of Ask decisions.
  • Compare agreement rate by router category.
  • Compare agreement rate by recommended action type.
  • Identify examples where the AI was directionally right but too confident.
  • Identify examples where the AI asked too many or too few clarifying questions.
  • Identify examples where the AI preserved the user's original intent well.
  • Identify examples where the AI summary distorted or over-formalized the user's language.
  • Use rejected/overridden recommendations as prompt-improvement material.

Support Quality

  • Review accepted support answers for accuracy, tone, and completeness.
  • Watch for answers that are technically correct but emotionally mismatched.
  • Watch for answers that overpromise features not yet built.
  • Watch for answers that should have escalated sooner.
  • Track users who repeatedly ask low-quality, abusive, or incoherent questions as a support/admin concern, not as an automatic punishment.
  • Consider whether repeated low-quality input should influence support triage or trigger admin review.

Data Integrity And Privacy

  • Confirm user-submitted Ask text is treated as private account/support context.
  • Confirm Admin review surfaces expose only what is necessary for review.
  • Confirm support-ticket creation preserves the original user wording.
  • Confirm internal handoff notes are marked internal and not exposed to the user.
  • Confirm downstream records keep links to Ask without copying unnecessary sensitive detail.
  • Confirm rejected/overridden items remain available for analysis without becoming public roadmap commitments.

Cost Tracking

  • Confirm successful Ask AI calls create AI operation log rows.
  • Confirm operation type, model, token usage, source table, source id, and estimated cost are captured where available.
  • Confirm the Admin dashboard cost tile loads after the SQL migration is applied.
  • Compare estimated cost by workflow as usage grows.
  • Revisit pricing snapshots when OpenAI/provider pricing changes.
  • Watch for unusually expensive interactions that may indicate prompt/context bloat.

Future Interpreter Work

  • Add or refine a dedicated Support Assistant module for how-to answers.
  • Add a Wishlist/Feature Interpreter that preserves original wording and extracts desired outcome, pain point, app area, urgency, and source.
  • Add a Bug/Friction Interpreter that captures tried/expected/actual behavior.
  • Add a Workflow Feedback Interpreter for repeated confusion or process suggestions.
  • Add a limited Off-Topic handler that safely closes or routes unusual input.
  • Make prompts and thresholds dynamically editable from Admin.
  • Consider auto-creation only after real agreement data supports it.
  • Keep auto-routing rollback as status/link retraction rather than physical deletion.

Early Review Questions

  • Are users finding Ask naturally, or does it need gentler placement or copy?
  • Are users treating Ask as intended: open dialogue rather than a formal support form?
  • Are Admin review decisions fast enough, or is there still too much data in the way?
  • Are accepted answers genuinely good, or merely plausible?
  • Are recommendations producing useful downstream records without creating clutter?
  • Which category or interpreter should be improved first based on real disagreement data?
  • Does the system preserve the user's original language and dignity?
